Media Queries

@media Best Practices

Mobile First @media queries

Write min-width and min-height,

not max-width or max-height

.box {
  /* Mobile CSS */
}

@media (min-width: 48em) {
  .box {
    /* Decent size screen styles */
  }
}

@media (min-width: 62em) {
  .box {
    /* Huge screen styles */
  }
}
Don't write max-width/height, larger screen styles will override smaller screen styles
Don't have to work mobile first, just have to end up with mobile first code.

Group all styles for an element

/* Widget Styles */
.widget { }

@media (min-width: 768px) {
  .widget { }
}

/* Doo Dad Styles */
.doo-dad { }

@media (min-width: 768px) {
  .doo-dad { }
}

Separating styles by media query gets confusing and hard to maintain, group styles by what element.

Only be as specific as necessary


@media (min-width: 768px) and screen {
  /* Is screen necessary, do we want to exclude print? */
  .widget { }
}

@media (min-width: 1024px) and (min-height: 768px) {
  /* Is height necessary, should 750px tall screens be excluded?  */
  .widget { }
}

Media queries should be based on content, not devices

  • Are columns squished?
  • Line lengths too long or short?
  • Images need to be bigger?

Devices can be a decent guide.
Test from 300px - 1600px wide.

CSS Layout in 2018

Document Flow & You

Going to try and explain the intention behind CSS
I'm going to take small liberties and make up some terms.
Document flow is how elements want to layout on the page, given their styles.
See text flow
See floated image
This is DF
They're cool
Want you to be able to see all the content
Read all the text
Tries not to let things overlap
Doesn't dig pixel perfection much
Don't be mean to document flow, respecting DF pays off.

Text Flow

  • How text behaves and tags like <strong> & <em>
  • Inline* elements mimic text flow
  • White space matters* (*up to one space!)
With no CSS, HTML will fall back to a textual flow
Doesn’t care about widths, heights, margins, padding, etc

Layout Flow

  • White space agnostic
  • Properties like:
    • Grid
    • Flexbox
    • Table
    • Columns
Composing major parts of doc
The large squares/rectangles that the web is good at
Content area might be flexbox (Layout flow), but inside of the left column we go back to text flow
This angry fellow represents properties that break/mess with document flow
It's not wrong to break document flow, it is gutsy
He'll do exactly what you say, even when it doesn't make sense
Lose the safeguards of document flow that made sure content is legible
He's been grumpy about all these new-fangled screen sizes
Test your CSS on many screen sizes, px densities, and browsers when you bring out this guy

Document Flow Breakers:

Document Flow

position: absolute, fixed, sticky

These will behave very rigidly and can cause overlap or bugs on differently sized screens.
Position fixed is particularly grumpy about smaller sized screens and zooming

Mess with Document Flow

  • position: relative
    (when top, right, bottom or left are set)
  • transform
  • negative margin or padding
These can cause overlap and can get a little weird
To get a modal like this, you have to break document flow
But notice inside of it, we're back to regular document flow

TL:Didn't think your cartoons were funny

  • Breaking the flow can cause complicated, hard to solve problems.
  • Try to use Flow-friendly tools first.
  • If you enjoy sanity, the large majority of the elements on the page should be flow-friendly.

Block Layout & Box Model

Last stop on the layout train!

Block Layout

The most basic and oldest layout method on the web.

  • A <div> has one default style, display: block
  • Block elements gets stacked on top of each other
  • They default to 100% width
  • Works well with all the box model properties:
    • width, height
    • margin, padding
    • border

When should I use block layout?

  • Nothing cooler makes sense
  • You want to use box model properties
  • The element shouldn't be inline with text
    (unlike <strong>, <em>, <span>)
